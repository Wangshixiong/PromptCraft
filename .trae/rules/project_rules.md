1. 项目核心目标 : 开发一款浏览器插件，用于存储、分类和快速使用提示词。这是我们所有架构设计的出发点。

2. 核心架构 - 模块化设计: 项目必须采用模块化思想。将 UI (用户界面)、Logic (业务逻辑) 和 Data (数据管理) 三个核心部分清晰分离，确保它们可以独立开发、修改和测试。

3. 数据层抽象: 必须创建一个独立的数据管理模块。该模块的接口设计要保持稳定，无论底层数据是来自本地存储 (初期) 还是云端服务器 (未来)。其他模块只调用接口，不关心数据具体存在哪里。

4. UI组件化: 所有用户界面元素（如提示词列表、编辑器、标签云）都必须构建为可复用的组件。这能极大提高开发效率，并方便未来统一修改样式或功能。

5. 可扩展性优先: 所有功能设计和编码时，必须优先考虑未来的扩展性。例如，数据模型应预留扩展字段；功能应通过配置来启用或禁用，而不是在代码里写死。

6. 分阶段开发路线图: 项目遵循迭代开发。我们明确以下阶段，架构设计必须能支撑这一演进：

- v1.0 (个人版): 实现所有核心功能，数据本地存储。
- v2.0 (同步版): 引入用户系统和云同步功能。
- v3.0 (协作版): 增加团队协作和提示词共享功能。

7. 统一配置管理: 创建一个全局配置文件，用于管理应用中的常量、功能开关和未来可能的API地址。禁止在业务代码中出现“魔法数字”或硬编码的字符串。

8. 版本控制规范: 所有代码变更必须通过 Git 进行管理。每个独立功能或修复都在一个新的分支上开发，完成后合并到主干，确保主干代码永远是稳定可用的。

9. 代码质量与注释: 所有代码都必须有清晰的注释，特别是核心模块的接口和复杂逻辑部分。AI生成的代码也必须遵守此规则，以方便您理解和后续维护。

10. 沟通要求: 当你提供技术方案或代码时，必须围绕以上架构规则展开，并使用简单易懂的语言解释“为什么这么设计”，以及它如何支持未来的扩展。

11. 当用户提出bug时，你可以通过搜索引擎查询相关问题，并借鉴搜索答案中的解决方案。

12. 修改代码，必须保持最小化修改原则。当涉及多个功能模块时，依次修改每个模块，确保每个修改都是为了解决特定问题而不是引入新问题。

13. 测试优先: 每个新功能或代码变更都必须经过充分测试。确保所有测试用例都能覆盖到，并且在回归测试中没有发现新的问题。

14. 文档更新: 每次代码变更或功能添加，都要及时更新项目文档。这包括但不限于修改README文件、API文档和用户手册（如果没有创建，如果有，就更新）。
